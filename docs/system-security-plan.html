Authorization Boundary

The authorization boundary encompasses all AWS resources provisioned by the Terraform IaC within a single AWS account and region, including:

  • Compute: Single EC2 instance (Amazon Linux 2023) running GitLab CE
  • Networking: VPC with public/private subnets across 2 AZs, NAT Gateway, VPC endpoints, security groups
  • Load Balancing & Edge: Application Load Balancer (ALB) with AWS WAF
  • Storage: EBS volumes (root + data), S3 buckets (backups, logs, CloudTrail, Config)
  • Secrets: AWS Secrets Manager secrets with CMK encryption
  • Encryption: 3 AWS KMS Customer Managed Keys (general, CloudTrail, EBS)
  • Monitoring: CloudTrail, CloudWatch, SNS
  • Security Services: GuardDuty, Security Hub, AWS Config, Amazon Inspector
  • Automation: Lambda functions (inactive user deactivation, secrets rotation, CISA KEV monitoring)
  • Identity: IAM roles/policies, ACM certificates

Excluded from boundary: Cloudflare DNS (external), end-user workstations, AWS management plane (covered by AWS FedRAMP P-ATO).

2. System Description / Purpose

Purpose

GitLab CE on AWS (IL2) provides a self-hosted source code management, CI/CD, and collaboration platform for Department of Defense personnel and contractors. The system hosts non-CUI public and low-sensitivity CUI source code repositories, issue trackers, and CI/CD pipelines on infrastructure aligned with DoD Impact Level 2 compliance requirements.

Functional Description

  • Source Code Management: Git-based version control with branch management, merge requests, and code review workflows
  • Issue Tracking: Project management with issues, milestones, and boards
  • CI/CD Pipelines: Automated build, test, and deployment pipelines
  • Container Registry: Docker image storage and distribution
  • Wiki & Documentation: Project-level wikis and documentation hosting

User Base

  • Total Users: Admin-managed accounts only; self-registration disabled
  • User Types: Administrators, Developers, Maintainers, Guests
  • Authentication: GitLab native authentication with mandatory TOTP-based 2FA
  • Access Method: HTTPS (web UI and Git over HTTPS with PATs); no SSH access

Separation of Duties

  • Infrastructure changes require Terraform plan review and explicit apply. Git history provides full audit trail.
  • GitLab admin operations are limited to designated administrators. Signup is disabled.
  • AWS access is separate from GitLab access. IAM follows least privilege. SSM provides auditable admin access.
  • Secret access is restricted via IAM and KMS key policies. Lambda functions have scoped permissions.

Access Termination

  • Accounts deactivated within [defined SLA] of personnel departure.
  • Automated deactivation of accounts inactive >90 days via Lambda (AC-2(3)).
  • SSM access revoked by removing IAM permissions.

Recovery Procedures (CP-10)

RTO: [To be defined]  |  RPO: 24 hours (daily backup cycle)

Full System Recovery

  1. Infrastructure Rebuild: terraform apply to recreate all resources. State in versioned, encrypted S3 backend.
  2. Data Restoration: Restore latest GitLab backup from S3 via gitlab-backup restore.
  3. Config Restoration: Download config backup from S3, extract to /etc/gitlab/.
  4. Secrets Restoration: Retrieve gitlab-secrets.json and root password from Secrets Manager.
  5. Reconfigure: gitlab-ctl reconfigure to apply restored configuration.
  6. DNS Update: Update Cloudflare DNS to point to new ALB endpoint.
  7. Verification: Confirm health check (/-/health), user access, and backup schedules.
Separate Data Volume: GitLab data (/var/opt/gitlab) is on a dedicated EBS volume, enabling independent snapshots, reattachment to replacement instances, and root volume replacement without data loss.
