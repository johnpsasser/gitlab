GitLab Quick Start Guide

Developer onboarding for the GitLab instance

Administrators: Before distributing this guide, replace gitlab.example.com throughout with your actual GitLab domain name.

1. Prerequisites

Before you begin, ensure you have:

  • A GitLab account (created by your administrator)
  • An authenticator app for two-factor authentication (e.g., Google Authenticator, Authy, 1Password)
  • Git installed on your machine

2. Connect to GitLab

  1. Open your browser and navigate to: 
    https://gitlab.example.com
  2. Enter your username and password provided by your administrator
  3. On first login, you'll be prompted to set up two-factor authentication (2FA)
  4. Scan the QR code with your authenticator app
  5. Enter the 6-digit code to verify, then save your recovery codes
First time? Your administrator will create your account and provide initial credentials. You'll be required to set up 2FA on your first login. Store your recovery codes in a secure location.

3. Create a Personal Access Token

Personal access tokens are used for Git operations over HTTPS and API access.

  1. Go to User Settings → Access Tokens
  2. Click "Add new token"
  3. Enter a Token name (e.g., "CLI access")
  4. Set an Expiration date (max 365 days)
  5. Select scopes:
    • api — full API access
    • read_repository — read-only Git access
    • write_repository — read-write Git access
  6. Click "Create personal access token"
  7. Copy the token immediately — it won't be shown again
Store your token securely. Use a password manager or your OS keychain. Never commit tokens to version control.

4. Clone Your First Repository

git clone https://gitlab.example.com/your-group/your-project.git
# When prompted for password, use your Personal Access Token

To avoid entering your token every time, configure Git credential storage:

# macOS (uses Keychain)
git config --global credential.helper osxkeychain

# Linux (stores in memory for 1 hour)
git config --global credential.helper 'cache --timeout=3600'

5. Troubleshooting

ProblemSolution
Can't reach GitLab URL Verify your internet connection. If the issue persists, check with your admin to confirm the service is running.
2FA setup not working Ensure your authenticator app's time is synced. Try a different authenticator app. If locked out, contact your administrator to reset 2FA.
Account locked after failed logins GitLab locks accounts after multiple failed attempts. Wait 10 minutes or contact your administrator to unlock.
HTTPS clone asks for username Use your GitLab username (not email). For the password, use your Personal Access Token.
SSL certificate error The GitLab instance uses a valid ACM certificate. If you see SSL errors, try clearing your browser cache or check with your admin.
"403 Forbidden" on git push Check that your Personal Access Token has the write_repository scope. Check that you have Developer or Maintainer role on the project.

6. Administrator: Post-Deployment Setup

The following steps are for system administrators after running terraform apply.

1 Populate GitLab Admin PAT

The user deactivation Lambda needs a GitLab admin Personal Access Token with api scope. Create one in GitLab, then store it:

aws secretsmanager put-secret-value \
  --secret-id gitlab/gitlab-admin-pat \
  --secret-string "glpat-xxxxxxxxxxxxxxxxxxxx"

2 Validate User Deactivation Lambda

The Lambda runs in DRY_RUN mode by default. Test invoke to verify it correctly identifies inactive users:

aws lambda invoke --function-name gitlab-user-deactivation /dev/stdout

Once validated, set dry_run = false in terraform.tfvars and re-apply.

3 Verify Inspector Scanning

Confirm Amazon Inspector is actively scanning the EC2 instance:

aws inspector2 list-coverage --filter-criteria '{
  "resourceType": [{"comparison": "EQUALS", "value": "AWS_EC2_INSTANCE"}]
}'

4 Verify ClamAV

Connect via SSM and verify ClamAV is running:

# Connect to the instance
aws ssm start-session --target <instance-id>

# Check ClamAV status
systemctl status clamd@scan
freshclam --version
ls -la /var/log/clamav/

5 Store Root Password for Rotation

The secrets rotation Lambda rotates the root password automatically every 90 days. Ensure the initial password is stored:

aws secretsmanager put-secret-value \
  --secret-id gitlab/root-password \
  --secret-string "your-initial-root-password"
Note: After the first automatic rotation, the password in Secrets Manager will be updated automatically. Always retrieve the current password from Secrets Manager before logging in as root.

6 Admin PAT Rotation (IA-5)

The GitLab admin PAT used by the user deactivation Lambda should be rotated periodically per organizational policy:

  1. Log into GitLab as the admin user
  2. Navigate to User Settings > Access Tokens
  3. Create a new token with api scope
  4. Update the secret in Secrets Manager: 
    aws secretsmanager put-secret-value \
  --secret-id gitlab/gitlab-admin-pat \
  --secret-string "glpat-NEW-TOKEN-VALUE"
  5. Revoke the old token in GitLab
  6. Test the Lambda to verify the new token works: 
    aws lambda invoke --function-name gitlab-user-deactivation /dev/stdout

7 gitlab-secrets.json Backup and Rotation

The gitlab-secrets.json file contains Rails secret keys and internal encryption keys. It is backed up to Secrets Manager but rotation requires careful coordination:

  1. Back up current secrets: 
    # Via SSM session
aws secretsmanager put-secret-value \
  --secret-id gitlab/secrets-json \
  --secret-string "$(sudo cat /etc/gitlab/gitlab-secrets.json)"
  2. Note: Rotating the Rails secret_key_base invalidates all existing user sessions and encrypted tokens. Plan for user re-authentication.
  3. After any rotation, run gitlab-ctl reconfigure and verify GitLab functionality.
Important: gitlab-secrets.json rotation is a manual procedure due to its impact on user sessions and encrypted data. Automate only after thorough testing in a non-production environment.
